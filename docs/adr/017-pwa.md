# ADR-017 — PWA / Installable Offline App

**Status:** Accepted (2026-06-20)
**Context feature:** F33 — PWA / Installable Offline App

## Context

PROJECT_SPEC calls for an "offline-ready architecture." Data is already offline
(Dexie/F10), but the app was not installable and the shell did not load offline.
Constraints: stay local-first, no new infra, minimal deps, don't break the F31
`html-to-image` path or the preview/capture/embed Web Workers, and Next 16 builds
with **Turbopack**.

## Decision

- **Hand-rolled `public/sw.js`, not Serwist/next-pwa.** Serwist's `withSerwist`
  is a webpack-config wrapper — it conflicts with the Turbopack build and adds a
  dependency. A plain same-origin service worker is Turbopack-safe, dependency-
  free, and CSP-safe (`default-src 'self'`). Its **network-first navigation**
  strategy means a fresh deploy's HTML always wins online, side-stepping the
  stale-hashed-chunk class of bug. Trade-off: coarser offline than a precache
  manifest — acceptable because Dexie data is already offline.

- **Pure `swStrategyFor` + a mirrored SW.** A plain SW can't import app modules,
  so the routing decision lives in a pure, unit-tested `lib/pwa/sw-strategy.ts`
  and `sw.js` duplicates it. The test pins the contract so the two can't silently
  diverge. Rules: passthrough for non-GET / cross-origin / `/api/*` / `*.worker.*`
  → F31 + the 3 workers are never intercepted; network-first for navigations;
  stale-while-revalidate for `/_next/static` + icons + manifest.

- **SVG manifest icons, not a binary PNG set.** Modern browsers accept SVG for
  installability; matches the existing gradient-square logo; no image tooling.

- **Production-only registration.** Dev keeps Turbopack/HMR unbroken (no SW). The
  registrar is a client island; `registerServiceWorker` takes an injected
  navigator so it unit-tests without a real SW runtime.

- **Plain info update toast, no action button.** The existing `ToastAction` is
  bookmark-specific (`intent: "undo" | "view"`, `payload: BookmarkId`). Rather
  than extend the toast intent system for one "Reload" action, the update toast
  is a plain info toast ("Refresh the page to update"); the new SW already
  `skipWaiting`'d, so any reload activates it.

- **`themeColor` in the `viewport` export**, not `metadata` (App Router requirement
  in Next 16).

## Consequences

- LinkNest is installable (Add to Home Screen) and its shell loads offline; a
  visited dashboard works fully offline (cached shell + Dexie data), and a cold
  start with no cache shows `/offline`.
- **Updating the SW** = bump the cache name (`linknest-shell-v1` → `-v2`); the
  `activate` handler purges old `linknest-shell-*` caches and `skipWaiting` +
  `clients.claim` roll it out; the update toast prompts a refresh.
- The SW only runs in production over HTTPS/localhost — the browser-smoke gate
  verifies registration on `npm run start`; SW control takes effect on the second
  load.
- If richer offline (full precache of all build assets) is ever needed, revisit
  Serwist once it has stable Turbopack support — the `swStrategyFor` contract and
  registrar would carry over.
